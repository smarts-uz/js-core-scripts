# Contributing

Thanks for your interest in improving **js_ai_category** — a Windows-focused,
Node.js (ESM) toolkit for processing Office and Markdown documents.

## Prerequisites

- **Node.js ≥ 20** (the repo pins a version in [`.nvmrc`](.nvmrc); run `nvm use`).
- **Windows** for the COM-driven operations (Word/Excel/PowerPoint automation via
  `winax`). Pure-logic, filesystem, Markdown, and YAML operations run anywhere.
- `npm install` to set up dev tooling (ESLint, Prettier, Husky, lint-staged) and
  the Husky pre-commit hook.

## Project layout

| Path                        | What                                                                                                |
| --------------------------- | --------------------------------------------------------------------------------------------------- |
| `utils/`                    | The library — one class per file (symlinked into a shared source repo).                             |
| `runs/<Class>/<method>.mjs` | One **standalone runner** per public static method, generated by `runs/_generate.mjs`.              |
| `runs/_*.mjs`               | Tooling: `_generate` (runners), `_gen-readme` (README), `_wire-shell` / `_wire-launch` (launchers). |
| `shell/`                    | Windows right-click context-menu launchers (point at runners).                                      |
| `.vscode/launch.json`       | VS Code debug configs (point at runners).                                                           |
| `tests/`                    | Jest suite — one spec per class (`tests/<Class>.test.js`).                                          |
| `config.yml`                | Non-secret runtime config. Secrets live only in a gitignored `.env`.                                |

## Adding or changing a `utils/` method

A public (non-`_`-prefixed) method must be wired across every surface **in the
same change**:

1. `npm run generate:runners` — (re)creates `runs/<Class>/<method>.mjs`.
2. Add/adjust the `shell/` launcher line and `.vscode/launch.json` config
   (`npm run generate:shell` / `npm run generate:launch` after editing their tables).
3. `npm run generate:readme` — refreshes `README.md` (it is **auto-generated**;
   edit the JSDoc in `utils/<Class>.js`, never `README.md` by hand).
4. Extend `tests/<Class>.test.js` with the new method's tests.

## Before opening a PR

```bash
npm run lint          # ESLint
npm run format:check  # Prettier
npm test              # Jest
```

The Husky pre-commit hook auto-runs ESLint `--fix` + Prettier on staged files
(`lint-staged`). Bypass with `git commit -n` only when truly necessary.

## Commit messages

Use **Conventional Commits** (`feat:`, `fix:`, `docs:`, `refactor:`, `test:`,
`chore:`, …). One logical change per commit. The [CHANGELOG](CHANGELOG.md) is
derived from this history.

## Secrets

Never put a credential in `config.yml`, source, or any tracked file — only in a
gitignored `.env` (see `INTEGRATIONS.md` for the variable names), read via `utils/Secrets.js`.
